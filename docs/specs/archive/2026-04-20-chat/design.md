## Context

파일 업로드·색인 파이프라인(`files`, `file_chunks`, pgvector 1536차원)은 이미 구현되어 있다. `get_current_user` 의존성과 `get_indexed_chunks(db, user_id)` 내부 인터페이스를 재사용할 수 있다. 현재 `app/api/v1/`에는 auth, files 라우터만 존재하며 chat 라우터가 없다.

## Goals / Non-Goals

**Goals:**
- 인증된 사용자가 자신의 색인 완료 문서만을 근거로 스트리밍 RAG 답변을 받을 수 있다.
- 세션 단위로 대화 이력(최근 10턴)이 유지되며 후속 질문에 반영된다.
- 스트림에 token → sources → done 이벤트 순서가 보장된다.
- 모든 세션·메시지·검색은 JWT sub 기준으로 격리된다.

**Non-Goals:**
- 타 사용자와의 세션 공유.
- 세션 제목 수동 편집 API.
- 스트리밍 중단 시 부분 답변 보존(이번 버전: 폐기).
- LLM 응답 피드백 수집·재랭킹.
- 동시 질문 직렬화(이번 버전: 허용, 경쟁 조건은 나중에).

## Decisions

### 1. LangGraph를 RAG 파이프라인 오케스트레이터로 사용

LangChain LCEL 체인만으로도 RAG가 가능하지만, 향후 에이전트 확장(검색 전략 분기, 재검색 루프)을 고려해 LangGraph StateGraph 구조를 채택한다. 현재 구현에서는 단순 선형 그래프(load_history → retrieve → generate)로 시작한다.

**이유**: 미래 확장 비용 최소화. 지금 LCEL로 구현하면 나중에 전면 재작성 필요.

### 2. SSE 이벤트 형식 — token / sources / done 3종

스트림 종료 직전에 sources 이벤트를 한 번 전송하고, 이후 done 이벤트로 마무리한다. 본문 중간에 인용 번호를 삽입하지 않는다.

**이유**: 클라이언트가 답변 텍스트와 출처 목록을 독립적으로 렌더링할 수 있어야 한다. 중간 삽입 방식은 텍스트 파싱 복잡도를 높인다.

### 3. 세션 제목 — 첫 메시지 앞 20자 슬라이싱

LLM을 별도 호출해 제목을 요약하지 않는다. 첫 메시지 content의 앞 20자를 그대로 사용한다.

**이유**: 추가 LLM 호출은 비용·지연을 발생시킨다. 현 단계에서 제목 품질 개선보다 구현 단순성이 우선이다.

### 4. 대화 이력 — 최근 10턴(user+assistant 쌍 기준)

슬라이딩 윈도우로 최근 20개 메시지(user 10 + assistant 10)를 LLM 프롬프트에 포함한다. 토큰 예산 압축은 이번 버전에서 구현하지 않는다.

**이유**: 대부분의 개인 문서 질의는 10턴 안에 맥락이 충분하다. 토큰 예산 슬라이딩은 모델별 컨텍스트 윈도우 계산이 필요해 복잡도가 높다.

### 5. pgvector 검색 — user_id 메타데이터 필터 + K=5

`get_indexed_chunks(db, user_id)`를 경유해 사용자별 격리를 강제한다. K=5는 3072토큰 내외를 컨텍스트로 확보한다.

**이유**: 금지 패턴 "유저 간 벡터 스토어 공유 금지" 준수. K=5는 정밀도와 컨텍스트 길이의 절충점이다.

### 6. 타인 세션 접근 — 404 통일

403 대신 404를 반환해 세션 존재 여부 자체를 노출하지 않는다. file-management와 정책 일치.

**이유**: 403은 "리소스가 존재하지만 권한이 없음"을 암시해 정보를 노출한다.

### 7. 스트리밍 시작 후 LLM 실패 — error 이벤트

`StreamingResponse`가 이미 열린 뒤에는 HTTP 상태코드를 바꿀 수 없다. SSE `event: error` 이벤트로 오류를 전달하고 부분 답변은 폐기(DB 미저장)한다.

**이유**: 스트리밍 시작 후 에러 처리를 위한 표준 SSE 패턴.

### 8. 색인 청크 없음 — 스트리밍으로 안내 메시지 반환

RAG 파이프라인 진입 전 `get_indexed_chunks` 결과가 빈 경우, LLM을 호출하지 않고 "참고할 문서가 없습니다" 메시지를 token 이벤트로 전송 후 done으로 종료한다. 해당 메시지는 assistant 메시지로 DB에 저장한다.

**이유**: 일반 지식 답변은 품질 보장 범위 밖(Non-goal). 빈 컨텍스트로 LLM을 호출하면 hallucination 위험이 있다.

## Risks / Trade-offs

- **LangGraph 학습 비용**: 초기 구현 난이도가 LCEL보다 높다 → 단순 선형 그래프로 시작해 복잡도를 점진적으로 높인다.
- **SSE 연결 유지**: 긴 LLM 응답 중 nginx/프록시 타임아웃 발생 가능 → ping 이벤트 주기적 전송 또는 인프라 레벨 타임아웃 조정이 필요하다(현재 버전 미구현, 알려진 한계).
- **동시 질문 경쟁 조건**: 동일 세션에 두 질문이 동시에 들어오면 두 어시스턴트 메시지가 뒤섞일 수 있다 → 이번 버전에서는 허용하되 향후 세션 레벨 락 적용 예정.
- **OpenAI 비용 가시성 없음**: 사용자에게 토큰 사용량이 노출되지 않는다 → 현재 Non-goal로 분류.
