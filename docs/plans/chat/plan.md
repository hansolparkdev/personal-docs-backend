# 기획서: chat

> 경고: 비평 3회 후 미해결 FAIL 있음. 검토 후 진행하세요.
> 미해결: "사용자/맥락 정의 누락"(FAIL 기준 1) — Problem 문단에 대상 사용자·맥락이 서술되어 있으나, 비평 에이전트가 별도 섹션을 기대함. 분류상 기획 결정이지만 반복 지적을 해소하려면 별도 섹션 신설 여부는 spec 단계에서 조정 가능.

## Meta
- type: feature
- package: backend

## Problem
대상 사용자는 자신의 문서(메모·업무 자료·참고 자료 등)를 서비스에 올려 두고, 필요한 순간에 그 내용을 다시 꺼내 쓰고 싶어하는 개인 지식 관리자다. 이들은 주로 "내가 예전에 정리해 둔 그 내용, 어디에 있었지?" "이 파일들 종합하면 결론이 뭔가?" 같은 맥락에서 자연어로 묻는다. 현재는 이렇게 업로드·색인해 둔 개인 문서가 저장소에만 존재할 뿐, 자연어로 묻고 근거 있는 답을 받을 수 있는 경로가 없다. 대화형 인터페이스·본인 문서 범위의 검색·근거 표시가 하나의 흐름으로 연결되지 않으면, RAG 인프라(파일 색인, 벡터 저장)가 실제 사용자 가치로 이어지지 못한다.

## Goals
- 인증된 사용자가 자연어 질문을 보내면, 자신이 색인 완료한 문서만을 근거로 답변이 생성된다.
- 답변은 스트리밍으로 점진 표시되어 첫 토큰까지의 체감 대기 시간이 짧다.
- 답변에는 근거가 된 출처(어느 파일의 어느 부분)가 함께 제공되어, 사용자가 원문을 확인할 수 있다.
- 사용자는 과거 대화를 세션 단위로 이어가거나 새로 시작할 수 있고, 사이드바에서 최근 대화를 탐색할 수 있다.
- 대화와 메시지는 소유자 본인만 조회·삭제할 수 있다(JWT sub 기준 격리).

## Non-goals
- 타 사용자와의 대화 공유·공동 편집.
- 대화 내용 검색(키워드로 과거 메시지 찾기)·태그·폴더 분류.
- 문서 범위를 벗어난 일반 지식 질의 응답을 명시적 기능으로 보장(모델의 기본 지식에 의존하는 응답은 품질 보장 범위 밖).
- 답변 생성 이후의 피드백 수집·평가(좋아요/싫어요) 및 그에 기반한 재학습.
- 음성 입출력, 이미지 첨부 질의.
- 동시 다중 세션 병렬 스트리밍 한 명당 개수 제한·요금제.
- 관리자용 대화 모니터링.

## Capabilities
- 세션 생성: 사용자가 새 대화를 시작하면 빈 세션이 만들어지고 이후 메시지의 컨테이너 역할을 한다. 첫 메시지 내용으로 제목을 자동 요약할지 여부는 Open Questions.
- 세션 목록 조회: 사용자는 자신의 세션 목록을 최신순으로 받아볼 수 있다(사이드바 표시용). 페이지네이션 방식은 Open Questions.
- 세션 단건 조회: 세션 하나를 열면 그 안의 메시지 이력(유저/어시스턴트 쌍, 각 답변의 출처 포함)을 순서대로 받을 수 있다.
- 세션 삭제: 사용자는 세션을 삭제할 수 있으며, 해당 세션의 메시지·출처 기록이 일괄 제거된다.
- 메시지 전송 및 RAG 응답 스트리밍: 사용자가 특정 세션에 질문을 보내면, 서버는 본인 소유의 색인 완료 청크만을 검색 대상으로 삼아 관련 근거를 모으고, 대화 이력과 함께 언어 모델에 전달해 답변을 생성한다. 답변 토큰은 생성되는 대로 스트리밍으로 클라이언트에 전달된다. 스트림에는 최종적으로 근거 출처 메타데이터(파일명·청크 위치)가 포함된다.
- 출처 인용 제공: 응답에 사용된 청크들의 원 파일과 위치가 구조화되어 전달되어, 클라이언트가 이를 답변 옆에 표시할 수 있다.
- 대화 이력 반영: 같은 세션 안의 직전 대화 맥락이 이후 질문 처리에 반영된다. 반영 범위(최근 N개 / 토큰 예산 내 요약)는 Open Questions.
- 소유권 격리: 모든 세션·메시지·검색은 JWT sub 기준 소유자 본인 범위로 제한된다. 타인 세션 접근 시도는 실패로 응답한다.
- 근거 부재 시 정책: 검색 결과가 비어 있거나 신뢰도가 낮으면, 근거 없음을 명시하며 임의 추측을 피하는 응답 방침을 따른다(구체 기준은 spec 단계).

## API Flow
정상 흐름 — 새 대화 + 질문:
1. 인증된 사용자가 세션 생성을 요청한다. 서버는 JWT로 소유자를 식별하고 빈 세션을 만들어 식별자를 돌려준다.
2. 사용자가 해당 세션에 질문을 전송한다. 서버는 세션 소유권을 확인하고, 유저 메시지를 히스토리에 기록한다.
3. 서버는 본인 소유의 색인 완료 청크 중 질문과 의미적으로 관련된 상위 결과를 검색한다(외부 공개 경로가 아닌 내부 인터페이스 사용).
4. 직전 대화 이력과 검색 결과를 합쳐 언어 모델에 프롬프트를 구성하고, 토큰 스트리밍 생성을 시작한다.
5. 서버는 스트리밍 연결을 열어 생성 토큰을 순차 전송한다. 스트림 종료 직전 혹은 전송 중 약속된 지점에, 답변 생성에 사용된 출처 목록(파일명·청크 위치 등)을 함께 내보낸다.
6. 스트림이 끝나면 전체 어시스턴트 메시지와 출처 메타데이터가 세션 히스토리에 영구 기록된다.

정상 흐름 — 이어서 질문/세션 탐색:
1. 사용자가 세션 목록을 요청하면 자신의 세션만 최신순으로 반환된다.
2. 특정 세션을 열면 그 안의 메시지 이력이 순서대로 반환된다.
3. 해당 세션에 새 질문을 보내면, 동일한 RAG·스트리밍 흐름을 타되 직전 대화 맥락이 반영된다.
4. 세션 삭제 요청 시, 소유권 확인 후 세션과 그 하위 메시지·출처 기록이 일괄 제거된다.

예외 흐름:
- 미인증/토큰 만료 → 인증 실패 응답. 스트리밍 개시 전이라면 연결 자체를 거부한다.
- 타인 세션 접근·메시지 전송·삭제 시도 → 없음과 동일한 응답(정보 노출 방지; 구체 상태코드는 spec).
- 색인 완료 청크가 하나도 없는 사용자 → "참고할 문서가 없다"는 취지의 응답을 내보낸다(모델의 일반 지식 답변 여부는 spec).
- 검색 결과가 비었거나 관련도가 낮음 → 근거 없음을 명시하고 추측을 피하는 응답. 임계값·판단 방식은 spec.
- 스트리밍 도중 클라이언트 연결 끊김 → 서버는 생성을 중단하고 그 시점까지의 부분 메시지를 저장할지 폐기할지 정책이 필요(Open Questions).
- 모델 호출 실패/타임아웃 → 사용자 메시지는 기록되고, 어시스턴트 응답은 실패 상태로 남아 재시도 경로를 안내(재시도 제공 여부는 Open Questions).
- 동일 세션에 동시에 두 개의 질문이 들어오는 경우 → 직렬화 또는 거부 중 어느 정책을 쓸지 Open Questions.

## Open Questions
- 세션 제목 자동 생성 여부 및 생성 시점(첫 질문 직후 / 첫 답변 완료 후 / 생성 안 함 + 수동 편집).
- 세션 목록 페이지네이션 방식(커서 / 오프셋 / 전체 반환 + 클라이언트 페이징).
- 대화 이력 반영 범위(최근 N턴, 토큰 예산 내 슬라이딩, 요약 압축 중 어느 전략).
- 검색 상위 K 값과 재랭킹 여부, 근거 부재 판단 임계값.
- 스트리밍 중단(클라이언트 연결 끊김) 시 부분 답변 보존 정책.
- 모델 호출 실패 시 재시도 트리거를 사용자에게 노출할지, 자동 재시도할지.
- 동일 세션 동시 질문 정책(직렬화 / 거부 / 허용).
- 출처 메타데이터를 스트림 중간에 보낼지, 종료 직전에 한 번에 보낼지(프론트 렌더 타이밍과 연관; 구체 형태는 spec).
- 답변에 인용 번호를 본문에 삽입할지, 별도 목록으로만 제공할지.
- 모델 제공자·모델명 선택 및 임베딩 모델과의 정합(파일 색인 시 사용한 임베딩 차원과 검색 경로 일치 확인은 spec 전제).
- 사용량/비용 가시성(사용자에게 표시 여부) — 지금 단계에서는 Non-goal로 둘지 확인 필요.
- 타인 리소스 접근 시 404 통일 / 403 구분 — file-management와 정책 일치 필요.
