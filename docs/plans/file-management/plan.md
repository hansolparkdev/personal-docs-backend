# 기획서: file-management

## Meta
- type: feature
- package: backend

## Problem
개인 문서를 기반으로 한 RAG 챗이 동작하려면, 사용자가 자신의 문서를 서비스에 올리고 검색 가능한 형태로 보관할 수 있어야 한다. 현재 시스템에는 사용자별 문서를 수용하고, 원본을 안전하게 보관하면서 동시에 의미 검색용 벡터로 색인하는 경로가 없다. 이 경로가 없으면 RAG 챗 기능은 아무 근거 없이 답하게 되므로, 파일 관리가 선행 기반이 된다.

## Goals
- 인증된 사용자가 자신의 파일을 업로드·목록 조회·개별 조회·삭제할 수 있다.
- 업로드된 파일은 소유자 본인만 접근·삭제할 수 있다(JWT sub 기준 격리).
- 업로드 직후 원본은 객체 스토리지에 보관되고, 텍스트가 추출 가능한 파일은 검색용 임베딩이 생성되어 RAG 챗에서 바로 활용 가능한 상태가 된다.
- 파일 삭제 시 원본과 관련 임베딩이 함께 제거되어, 삭제 후에는 RAG 응답에 해당 문서 내용이 더 이상 등장하지 않는다.
- 업로드 결과(성공 / 처리 중 / 실패)를 사용자가 조회를 통해 확인할 수 있다.

## Non-goals
- 파일 공유, 공동 편집, 권한 위임(다른 사용자에게 읽기 권한 부여).
- 파일 내용 수정(파일은 불변; 교체는 삭제 후 재업로드).
- 폴더/태그/버전 관리.
- 이미지·오디오·영상의 내용 기반 검색(텍스트 추출이 불가능한 포맷은 원본 보관만).
- 파일 내용의 OCR 및 복잡한 표/수식 구조 추출.
- 관리자용 전체 파일 조회·관리 화면.
- 업로드 용량/요금제 관리.

## Capabilities
- 파일 업로드: 사용자가 자신의 파일을 업로드하면, 원본은 객체 스토리지에 저장되고 메타데이터가 기록된다. 동일 사용자 내에서 어떤 키로 중복을 판단할지는 Open Questions.
- 텍스트 추출 및 임베딩: 지원 포맷의 파일은 업로드 이후 텍스트가 추출·분할되어 벡터 임베딩으로 색인된다. 추출 실패 시에도 원본은 유지되고 상태로 구분된다.
- 파일 목록 조회: 사용자는 자신이 업로드한 파일 목록과 각 파일의 처리 상태(원본 저장 완료 / 색인 진행 / 색인 완료 / 색인 실패)를 확인할 수 있다.
- 파일 단건 조회: 사용자는 자신의 파일 메타데이터를 조회하고, 원본을 다시 내려받을 수 있다.
- 파일 삭제: 사용자는 자신의 파일을 삭제하며, 이때 원본·메타데이터·임베딩이 모두 일괄 제거된다.
- 소유권 검증: 모든 파일 관련 요청은 JWT sub 기준으로 소유자 확인 후에만 처리된다. 타인 파일 접근 시도는 실패로 응답한다.
- RAG 질의용 공급 계약: RAG 챗 기능이 "이 사용자의 색인 완료된 청크들"을 조회할 수 있는 내부 경로가 존재한다(외부 공개 API는 아님; 구체 형태는 spec 단계에서 결정).

## UX Flow / API Flow
정상 흐름 — 업로드:
1. 인증된 사용자가 파일을 전송한다.
2. 서버는 JWT로 소유자를 식별하고, 파일 크기·포맷 허용 여부를 확인한다.
3. 원본을 객체 스토리지에 저장하고 메타데이터를 기록한다. 사용자에게는 수신 완료 응답이 돌아간다.
4. 이어서 텍스트 추출·분할·임베딩 색인이 진행된다(동기 또는 비동기 — Open Questions).
5. 처리 결과에 따라 파일 상태가 "색인 완료" 또는 "색인 실패"로 갱신된다.

정상 흐름 — 조회/삭제:
1. 사용자가 목록 또는 단건 조회를 요청하면, 서버는 자신 소유분만 반환한다.
2. 삭제 요청 시, 서버는 소유권을 확인한 뒤 원본·메타데이터·임베딩을 제거하고 성공 응답을 돌린다.

예외 흐름:
- 미인증/토큰 만료 → 인증 실패 응답.
- 타인 파일 접근·삭제 시도 → 없음과 동일한 응답(정보 노출 방지 — 구체 응답 정책은 spec).
- 지원하지 않는 포맷/초과 크기 → 업로드 거부.
- 텍스트 추출 실패 → 원본은 유지, 상태는 "색인 실패"로 표기, 사용자에게 재시도 경로 제공 여부는 Open Questions.
- 색인 진행 중 파일을 RAG 챗이 질의 → 해당 파일은 이번 응답의 근거에서 제외된다.

## Open Questions
- 지원할 초기 파일 포맷 범위(txt/markdown/pdf/docx 중 1차 범위).
- 업로드 크기 상한 및 사용자별 총 용량 상한.
- 텍스트 추출·임베딩을 요청 내 동기 처리로 할지, 비동기(백그라운드 작업 큐)로 분리할지.
- 중복 업로드 판단 기준(파일명 / 해시 / 없음).
- 원본 다운로드 방식(서버 경유 스트림 / 스토리지 사전서명 URL) — 보안과 UX 트레이드오프.
- 타인 리소스 접근 시 응답을 404로 통일할지 403으로 구분할지.
- 색인 실패 파일에 대한 재색인 트리거 제공 여부.
- 삭제 시 임베딩 제거를 트랜잭션으로 묶을지, 사후 정리 작업으로 허용할지(일관성 vs 성능).
- 청크 크기·오버랩·임베딩 모델 선택 등 구체 파라미터는 spec 단계에서 결정.
